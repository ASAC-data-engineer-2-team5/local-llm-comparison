# 사내 규정 챗봇 에이전트 프로젝트 개요

## 1. 프로젝트 목표

**최종 목표:** 사내 규정 챗봇 에이전트 구축

직원이 사내 규정, 정책, 절차 등을 자연어로 질문하면 정확하게 답변해주는 AI 에이전트 시스템.
외부 API(OpenAI 등)에 의존하지 않고, 온프레미스 환경에서 로컬 LLM을 운용하는 것을 목표로 한다.

---

## 2. 프로젝트 방향 및 전제 가정

### 핵심 방향

- **한정된 하드웨어 환경**에서 운용 가능한 최적의 모델 및 아키텍처를 탐색
- 비용 효율적이고 보안이 확보된 온프레미스 LLM 운용

### 사용 환경 가정

| 항목 | 내용 |
|------|------|
| 회사 규모 | 400~500명 |
| 평시 동시 접속자 | 약 10명 |
| 피크 시즌 예시 | 연말정산, 인사 평가 시즌 등 |

### 피크 시즌 시나리오 고려

- 연말정산, 취업규칙 개정, 인사 평가 시즌 등 특정 기간에 동시 접속자가 급증할 수 있음
- 평시 대비 3~5배 트래픽 증가 가능성 → 큐잉 전략, 모델 응답 캐싱, 배치 처리 방안 검토 필요

---

## 3. 개발 환경 (로컬 테스트 머신)

| 항목 | 사양 |
|------|------|
| 장치명 | DESKTOP-DKOJG5Q |
| CPU | 12th Gen Intel Core i7-12800H (2.40 GHz) |
| RAM | 32.0 GB (31.7 GB 사용 가능) |
| GPU | NVIDIA T600 Laptop GPU (**4 GB VRAM**) |
| 보조 GPU | Intel Iris Xe Graphics (128 MB) |
| 저장소 | 477 GB 중 약 304 GB 여유 |
| OS | Windows 11 Pro 64bit, x64 |

### 하드웨어 제약 포인트

- **GPU VRAM 4 GB** — 가장 핵심적인 병목. 대부분의 7B 이상 모델은 풀 정밀도(FP16) 로드 불가
- **RAM 32 GB** — CPU 오프로딩 시 활용 가능, 단 응답 속도 저하 감수 필요
- VRAM 한계를 고려해 **4-bit / 8-bit 양자화(quantization)** 또는 **GGUF 포맷** 모델 위주로 탐색

## 4. 예상 기술 스택

```
┌─────────────────────────────────────────────────────┐
│                  사용자 인터페이스                    │
│              (Slack / 웹 채팅 / 내부 포털)            │
└───────────────────────┬─────────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────────┐
│              오케스트레이션 레이어                    │
│           LangGraph (멀티 에이전트 워크플로우)        │
│       + 하네스 엔지니어링 (Prompt Harness / Guard)   │
└──────────┬────────────────────────┬──────────────────┘
           │                        │
┌──────────▼──────────┐  ┌─────────▼──────────────────┐
│    RAG 파이프라인    │  │       멀티 에이전트          │
│  - 문서 청킹 / 임베딩│  │  - 라우터 에이전트           │
│  - Vector DB 검색    │  │  - 규정 조회 에이전트        │
│  - 컨텍스트 주입     │  │  - 요약 / 재질문 에이전트    │
└──────────┬──────────┘  └─────────┬──────────────────┘
           │                        │
┌──────────▼────────────────────────▼──────────────────┐
│                   Vector DB                           │
│         (Chroma / Qdrant / Milvus 등 검토 중)         │
└───────────────────────┬──────────────────────────────┘
                        │
┌───────────────────────▼──────────────────────────────┐
│              로컬 LLM (온프레미스)                    │
│   Ollama / llama.cpp 기반 GGUF 양자화 모델 운용       │
│   (현재 비교 실험 진행 중 → 최적 모델 선정 목표)      │
└──────────────────────────────────────────────────────┘
```

### 기술 스택 상세

| 레이어 | 기술 후보 |
|--------|-----------|
| 오케스트레이션 | LangGraph |
| 에이전트 설계 | 멀티 에이전트 아키텍처 (Router + Specialist) |
| 프롬프트 관리 | 하네스 엔지니어링 (Prompt Harness) |
| RAG | LangChain RAG 파이프라인 |
| Vector DB | Chroma (로컬), Qdrant (프로덕션 검토) |
| 임베딩 모델 | 로컬 임베딩 모델 (e.g. nomic-embed-text, bge-m3) |
| LLM 런타임 | Ollama / llama.cpp |
| 모델 포맷 | GGUF (4-bit / 8-bit 양자화) |

## 5. 현재 단계: 모델 비교 실험

이 레포지토리의 현재 목적은 **로컬 LLM 모델 비교(local-llm-comparison)**다.

4 GB VRAM 제약 내에서 실제로 동작 가능하고 한국어 성능이 충분한 모델을 선별하기 위해 실험 중.

### 비교 기준 (예정)

- [ ] VRAM 4 GB 내 실제 구동 가능 여부
- [ ] 한국어 이해 및 생성 품질
- [ ] 응답 속도 (tokens/sec)
- [ ] 문서 기반 QA 정확도 (RAG 연동 시)
- [ ] 동시 요청 처리 능력 (10명 기준)

### 주요 검토 모델 (예시)

| 모델 | 파라미터 | 양자화 | VRAM 추정 | 한국어 지원 |
|------|---------|--------|-----------|------------|
| Qwen2.5-7B | 7B | Q4_K_M | ~4.5 GB | 양호 |
| EXAONE-3.5-7.8B | 7.8B | Q4_K_M | ~4.8 GB | 우수 |
| Gemma-3-4B | 4B | Q4_K_M | ~2.8 GB | 보통 |
| Phi-4-mini | 3.8B | Q4_K_M | ~2.5 GB | 보통 |
| Llama-3.1-8B | 8B | Q4_K_M | ~5.0 GB | 보통 |

> ※ VRAM 수치는 추정값이며 실측 필요. 4 GB VRAM 초과 모델은 CPU 오프로딩 병행 실험 예정.

## 6. 향후 로드맵 (대략)

```
Phase 1 (현재)  : 로컬 LLM 모델 비교 실험 → 최적 모델 선정
Phase 2         : RAG 파이프라인 구축 + Vector DB 연동
Phase 3         : LangGraph 기반 멀티 에이전트 설계
Phase 4         : 하네스 엔지니어링 적용 (Prompt Guard, 평가 루프)
Phase 5         : 사내 규정 문서 임베딩 + 챗봇 프로토타입
Phase 6         : 부하 테스트 + 피크 시즌 대응 전략 수립
```
